# ADR 003: Сервис удалённого выполнения скриптов (Script Execution Service)

Статус: proposed
Дата: 2026-03-08

## Контекст

Необходимо дать возможность администраторам и операторам запускать скрипты на инстансах backend-сервисов: миграции данных, диагностика, сбор метрик, горячая конфигурация и т.д.

Текущая система ролей:
- **Системный уровень:** `is_admin` (boolean) — полный доступ ко всем операциям управления пользователями.
- **Проектный уровень:** `owner` / `editor` / `viewer` — права в рамках проекта.

Проблема: `is_admin` — слишком грубый инструмент. Не каждый админ должен иметь право запускать произвольные скрипты на сервисах. Проектные роли вообще не подходят — выполнение скриптов не привязано к проекту.

## Решение

### 1. Расширение системы доступов: RBAC v2

Вместо точечного решения (таблица capabilities) — полноценная система ролей и разрешений, описанная в **[docs/rbac-v2-design.md](../rbac-v2-design.md)**.

Ключевое для script-service:
- Permissions: `scripts.execute`, `scripts.manage`, `scripts.view_logs`
- Системная роль **operator** включает все script permissions
- Superadmin имеет все permissions неявно
- Все действия аудируются через `audit_log`

### 2. Новый сервис: script-service (порт 8004)

Центральный сервис управления скриптами. Отвечает за:
- Хранение реестра разрешённых скриптов (не произвольный код — только зарегистрированные скрипты).
- Валидацию и диспетчеризацию команд на запуск.
- Отслеживание статуса выполнения.
- Хранение логов и результатов.

### 3. Script Runner — модуль в каждом сервисе

Лёгкий модуль, встраиваемый в каждый backend-сервис. Принимает команды от script-service и выполняет зарегистрированные скрипты в контексте сервиса.

## Не цели

- Произвольное выполнение кода (только зарегистрированные скрипты из реестра).
- Замена CI/CD пайплайнов.
- Выполнение скриптов на frontend или firmware.

## Альтернативы

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| SSH + Ansible | Проверенный подход | Не интегрировано с RBAC, нет аудита в платформе |
| Флаг `is_admin` для доступа | Минимум изменений | Слишком грубо, нарушает принцип наименьших привилегий |
| Отдельный API в каждом сервисе | Простая реализация | Дублирование логики, нет центрального контроля |

## Риски

- **Безопасность:** выполнение скриптов — привилегированная операция. Митигация: whitelist скриптов, аудит-лог, capabilities.
- **Стабильность:** скрипт может повредить сервис. Митигация: таймауты, изоляция через subprocess, ограничение ресурсов.
- **Связность:** script-service становится зависимостью всех сервисов. Митигация: коммуникация через RabbitMQ (асинхронно), сервис работает и без script-service.
